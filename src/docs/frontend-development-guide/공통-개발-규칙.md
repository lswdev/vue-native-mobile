## 1. 프로젝트 레이어 구조

본 프로젝트는 **4계층(Layer) 아키텍처**를 기반으로 개발한다.  
각 레이어는 단방향 의존성을 유지하며, 상위 레이어가 하위 레이어를 참조한다.

```
┌──────────────────────────────────┐
│        View Layer (표현)          │  pages/, layouts/, components/
│  Vue SFC · Tailwind · CSS        │
├──────────────────────────────────┤
│      Application Layer (응용)     │  stores/, composables/
│  Pinia · Vue Composables         │
├──────────────────────────────────┤
│      Service Layer (서비스)       │  services/, utils/
│  API 호출 · 비즈니스 로직         │
├──────────────────────────────────┤
│    Infrastructure Layer (기반)    │  api/, plugins/, router/
│  Axios · Router · 환경 설정       │
└──────────────────────────────────┘
```

### 레이어별 책임

| 레이어 | 폴더 | 역할 |
|--------|------|------|
| **View** | `pages/`, `layouts/`, `components/` | UI 렌더링, 사용자 이벤트 처리 |
| **Application** | `stores/`, `composables/` | 상태 관리, 재사용 비즈니스 로직 |
| **Service** | `services/`, `utils/` | API 조합, 도메인 로직 캡슐화 |
| **Infrastructure** | `api/`, `plugins/`, `router/` | HTTP 통신, 라우팅, 플러그인 초기화 |

> **원칙:** View → Application → Service → Infrastructure 순으로만 의존한다.  
> View에서 직접 `api/` 를 import 하는 것은 **금지**한다.

### Composable 작성 규칙

```typescript
// composables/usePagination.ts
import { ref, computed } from 'vue'

export interface PaginationOptions {
  initialPage?: number
  pageSize?: number
}

export function usePagination(options: PaginationOptions = {}) {
  const { initialPage = 1, pageSize = 20 } = options

  const currentPage = ref(initialPage)
  const totalCount = ref(0)

  const totalPages = computed(() =>
    Math.ceil(totalCount.value / pageSize)
  )

  function setPage(page: number): void {
    currentPage.value = Math.max(1, Math.min(page, totalPages.value))
  }

  // ✅ 반환값 명시적으로 구성
  return {
    currentPage: readonly(currentPage),
    totalCount,
    totalPages,
    setPage,
  }
}
```

## 6. Pinia 상태 관리 규칙

### 스토어 작성 기준

- **Composition API 스타일**(`defineStore` with setup function) 사용 통일
- 스토어 ID는 도메인명 기준 (`'auth'`, `'gis'`, `'energy'`)
- 스토어에서 직접 API 호출 **금지** → `services/` 레이어 통해 호출

```typescript
// stores/auth.store.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import { authService } from '@/services/auth.service'
import type { UserDto } from '@/types/api.types'

export const useAuthStore = defineStore('auth', () => {
  // State
  const user = ref<UserDto | null>(null)
  const token = ref<string | null>(null)
  const isLoading = ref(false)

  // Getters (computed)
  const isAuthenticated = computed(() => !!token.value)
  const userName = computed(() => user.value?.name ?? '')

  // Actions
  async function login(email: string, password: string): Promise<void> {
    isLoading.value = true
    try {
      const result = await authService.login({ email, password })
      token.value = result.token
      user.value = result.user
    } finally {
      isLoading.value = false
    }
  }

  function logout(): void {
    user.value = null
    token.value = null
  }

  return {
    // State (읽기 전용으로 노출)
    user: readonly(user),
    token: readonly(token),
    isLoading: readonly(isLoading),
    // Getters
    isAuthenticated,
    userName,
    // Actions
    login,
    logout,
  }
}, {
  persist: true, // pinia-plugin-persistedstate 사용 시
})
```

### 스토어 분리 기준

| 스토어 | 관리 대상 |
|--------|-----------|
| `app.store` | 전역 UI 상태 (테마, 사이드바, 알림) |
| `auth.store` | 인증 상태, 토큰, 사용자 프로필 |
| `gis.store` | 지도 상태, 선택된 레이어, 뷰포트 |
| `energy.store` | 에너지 데이터, 발전량 캐시 |

## 9. API 레이어 규칙

### Axios 인스턴스 설정

```typescript
// api/axios.ts
import axios, { type AxiosInstance, type AxiosError } from 'axios'
import { useAuthStore } from '@/stores/auth.store'
import router from '@/router'

export const apiClient: AxiosInstance = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 15_000,
  headers: {
    'Content-Type': 'application/json',
  },
})

// 요청 인터셉터
apiClient.interceptors.request.use((config) => {
  const authStore = useAuthStore()
  if (authStore.token) {
    config.headers.Authorization = `Bearer ${authStore.token}`
  }
  return config
})

// 응답 인터셉터
apiClient.interceptors.response.use(
  (response) => response,
  async (error: AxiosError) => {
    if (error.response?.status === 401) {
      const authStore = useAuthStore()
      authStore.logout()
      await router.push('/login')
    }
    return Promise.reject(error)
  }
)
```

### 서비스 레이어 작성 패턴

```typescript
// services/energy.service.ts
import { apiClient } from '@/api/axios'
import { API_ENDPOINTS } from '@/api/endpoints'
import type { ApiResponse, EnergyDataDto, EnergyQueryParams } from '@/types/api.types'

export const energyService = {
  async getGenerationData(
    params: EnergyQueryParams
  ): Promise<EnergyDataDto[]> {
    const { data } = await apiClient.get<ApiResponse<EnergyDataDto[]>>(
      API_ENDPOINTS.ENERGY.GENERATION,
      { params }
    )
    return data.data
  },

  async updateFacility(
    id: number,
    payload: Partial<FacilityDto>
  ): Promise<FacilityDto> {
    const { data } = await apiClient.patch<ApiResponse<FacilityDto>>(
      `${API_ENDPOINTS.FACILITY.BASE}/${id}`,
      payload
    )
    return data.data
  },
}
```

### 네비게이션 가드

```typescript
// router/guards.ts
import type { Router } from 'vue-router'
import { useAuthStore } from '@/stores/auth.store'

export function setupNavigationGuards(router: Router): void {
  router.beforeEach(async (to, _from) => {
    const authStore = useAuthStore()

    // 인증 필요 페이지 접근 제어
    if (to.meta.requiresAuth && !authStore.isAuthenticated) {
      return { name: 'login', query: { redirect: to.fullPath } }
    }

    // 이미 로그인된 사용자의 로그인 페이지 접근 차단
    if (to.name === 'login' && authStore.isAuthenticated) {
      return { name: 'home' }
    }

    // 페이지 타이틀 설정
    if (to.meta.title) {
      document.title = `${to.meta.title} | KIE-REMS`
    }
  })
}
```

## 13. 금지 사항

### 절대 금지 (🚫)

| 금지 항목 | 이유 |
|-----------|------|
| `any` 타입 사용 | 타입 안전성 파괴 |
| View에서 직접 `api/` import | 레이어 의존성 위반 |
| 스토어에서 직접 axios 호출 | 레이어 의존성 위반 |
| 인라인 스타일 (`style="..."`) | 유지보수 불가, 다크모드 미지원 |
| `!important` 남용 | CSS 우선순위 추적 불가 |
| `<script>` (Options API) | Composition API 통일 원칙 위반 |
| `localStorage` 직접 접근 | store 또는 유틸로 추상화 필요 |
| 하드코딩된 API URL | 환경 변수 사용 필수 |
| `v-html` 무분별한 사용 | XSS 취약점 |
| BEM 없는 구조용 클래스 (`header`, `wrapper` 단독) | 스타일 충돌·추적 불가 |
| BEM Element 중첩 (`block__a__b`) | 네이밍 규칙 위반 |
| 태그/후손 선택자로 UI 구조 스타일링 | 마크업 결합·특이도 증가 |

### 주의 사항 (⚠️)

| 주의 항목 | 대안 |
|-----------|------|
| `console.log` 개발 후 미제거 | ESLint 경고로 체크, `console.warn/error`만 허용 |
| Tailwind 임의 값(`w-[324px]`) 남용 | `tailwind.config.ts`에 토큰 등록 |
| 컴포넌트 200줄 초과 | 컴포넌트 분리 또는 Composable 추출 |
| 스토어 남용 | 지역 상태는 `ref()`로 관리, 전역 필요시에만 스토어 |
| `defineExpose` 남용 | 부모-자식 간 직접 메서드 호출 최소화 |
| Tailwind만으로 복잡한 컴포넌트 상태 표현 | BEM Modifier + scoped/CSS Modules로 구조·상태 분리 |
| Block 이름과 무관한 CSS 클래스 | 컴포넌트명 기반 Block 접두 사용 |
